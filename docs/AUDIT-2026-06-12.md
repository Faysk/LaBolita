# Auditoria técnica do LaBolita

Data: 12 de junho de 2026

## Resumo executivo

O projeto está em bom estado para continuar operando a Copa, com regras
críticas no PostgreSQL, RLS forte, smoke tests úteis, backup agendado e UX já
bem acima de um MVP comum. A produção respondeu ao smoke test e a configuração
de autenticação remota está saudável.

O principal problema encontrado nesta auditoria foi operacional: o check externo
**Supabase Preview** falhava porque `supabase/config.toml` ainda declarava a
chave obsoleta `api.auto_expose_new_tables`. A chave foi removida.

Também foram encontrados dois desvios pós-início da Copa: a fonte secundária da
agenda passou a omitir jogos já iniciados/finalizados, e a documentação ainda
dizia que o cron de resultados nunca pontuava automaticamente, apesar do código
já auto-finalizar jogos de fase de grupos após atraso de segurança.

## Correções aplicadas

- Removida a chave obsoleta `auto_expose_new_tables` de
  `supabase/config.toml`, corrigindo a causa do check vermelho do Supabase.
- Ajustado `scripts/verify-schedule-sources.mjs` para continuar validando os
  jogos futuros quando a fonte secundária deixa de listar partidas já passadas.
- Adicionado `RESULTS_AUTO_FINALIZE_DELAY_MINUTES` ao `.env.example`.
- Atualizada a documentação de resultados e administração para refletir a
  auto-finalização da fase de grupos e a confirmação manual do mata-mata.
- Recriado o workflow `Production smoke` com execução manual e agendada.
- Adicionado `permissions: contents: read` ao workflow principal de CI.

## O que está bom

- **Banco como fonte das regras:** `save_prediction`, `finalize_match`, cálculo
  de pontos, bloqueio por horário, entrada tardia, correções e ranking ficam no
  PostgreSQL.
- **Segurança:** RLS, RPCs com `security definer` e `search_path`, service role
  fora do browser, headers fortes, proxy de sessão Supabase e cron protegido por
  bearer secret.
- **Testes:** lint, typecheck, Vitest, smoke de banco PGlite, UI smoke,
  cobertura e production smoke estão funcionando.
- **Operação:** `/api/health`, sync de resultados com estado persistido,
  fallback de feed, backups 3 vezes ao dia por GitHub Actions e retenção de 7
  dias.
- **Produto:** home, palpites, bolões, rankings, Copa, seleções e jogadores já
  têm uma experiência coerente em desktop e mobile.

## Pontos críticos

### P0 - Corrigido nesta auditoria

- **Supabase Preview falhava na main.** Causa: chave removida do schema atual do
  Supabase CLI/Preview. Correção: remover `api.auto_expose_new_tables`.

### P1 - Antes de mudanças grandes em produção

- **Restore de backup precisa ensaio.** Backups existem, mas o processo de
  restore deve ser testado em homolog/staging antes de ser considerado plano de
  recuperação.
- **Auto-finalização exige monitoramento.** O cron pode pontuar jogos de grupo.
  Isso é desejado, mas precisa alerta em caso de `results_sync_state.status =
  error`, divergência de feed ou correção manual frequente.
- **Evitar refactors grandes durante rodada ativa.** Componentes centrais ainda
  são grandes; dividir é bom, mas deve entrar por PRs pequenos com preview e
  smoke remoto.

## Pontos de atenção

- **Páginas públicas podem ficar dinâmicas no build real.** Com variáveis
  Supabase configuradas, `next build` marcou várias páginas públicas como `ƒ`.
  Isso não quebra, mas custa cache/performance. Route groups públicos/autenticados
  e cache por request podem melhorar.
- **CSP ainda usa `'unsafe-inline'`.** Aceitável no momento, mas nonce/hash para
  scripts e estilos deve entrar antes de ampliar integrações externas.
- **CI pode endurecer mais.** O CI agora declara `permissions: contents: read`
  e o production smoke voltou. Próximo passo: pin de actions por SHA e alertas
  para falhas fora do GitHub.
- **Supabase lint local depende de Docker/Postgres.** Nesta máquina, `db:lint`
  não executa porque não há Supabase local em `127.0.0.1:54322`.
- **Componentes grandes:** `pools-workspace.tsx`, `admin-match-queue.tsx`,
  `competition-overview.tsx` e `match-card.tsx` merecem fatiamento gradual.
- **Limite de 104 partidas está espalhado.** É correto para a Copa atual, mas
  futuro multi-torneio deve centralizar essa configuração.
- **Fallback ESPN não tem SLA.** Deve continuar como contingência, não como
  garantia operacional.

## Bugs e inconsistências encontrados

- Fonte secundária de agenda passou a listar 102 partidas porque omitiu os dois
  primeiros jogos. O script agora aceita apenas ausências de jogos já iniciados.
- Documentação dizia que o placar observado nunca pontuava automaticamente; isso
  estava desatualizado diante da auto-finalização de jogos de grupo.
- O check Supabase Preview vermelho era causado por configuração obsoleta, não
  por migration ou bug de dados.

## Sugestões de features

- Alertas de production smoke em Slack/e-mail quando o workflow falhar.
- Painel operacional com última sync, erros recentes, jogos auto-finalizados,
  próximos bloqueios e correções administrativas.
- Alertas para `/api/health`, falha de OAuth, cron de resultados, erro frontend
  e divergência do feed.
- Drill de restore automatizado em homolog.
- Notificações de palpites pendentes e lembretes antes do bloqueio.
- Estatísticas de atletas a partir de provedor com SLA: gols, assistências,
  cartões, minutos e rankings por seleção. Manter dados estáticos de elencos
  como base, mas não misturar estatística ao vivo sem fonte confiável.
- Evolução do ranking por rodada/fase e histórico de viradas.
- PWA/offline somente leitura depois da estabilização operacional.

## Validação executada

- `npm run check`: passou.
- `npm run build`: passou.
- `npm run test -- --coverage`: passou.
- `npm audit --json`: 0 vulnerabilidades.
- `npm run schedule:verify:sources`: passou; fonte secundária listou 102/104
  linhas e omitiu 2 jogos já iniciados.
- `npm run test:production -- --allow-unconfigured-sync`: passou contra
  `https://labolita.faysk.dev`.
- `npm run auth:smoke:remote`: passou; Google OAuth configurado, 1 master admin
  e termos ativos.

## Limitações

- `npx supabase status --workdir supabase` não pôde chegar ao ambiente local
  porque Docker não está ativo/disponível nesta máquina.
- `npx supabase db lint --local --fail-on error` não pôde conectar em
  `127.0.0.1:54322`.
- Não executei smoke remoto que altera banco de produção nem forcei o cron de
  resultados manualmente nesta auditoria.
